The following recurring patterns were identified across the unclassified candidates. The `.dockerignore` pattern (using the ignore file to discover files expected but absent) is noteworthy as an auditor technique, not a defect class of its own. The three defect-class patterns above are promoted to manifest entries below.

## Proposed Manifest Additions

Three patterns promoted to manifest candidate status. Ready-to-paste YAML for `~/.claude/docs/standards-manifest.yaml`.

### FOUND-NEW-002: Post-migration artefact decay

Triggers when a frontend or backend stack is replaced. Stale references to the deprecated stack accumulate in Dockerfiles, docker-compose files, CORS origin lists, and utility scripts without any automated check to catch them.

```yaml
- id: FOUND-NEW-002
  domain: foundations
  severity: important
  description: >
    Post-migration artefact decay: references to a removed stack remain in
    Dockerfile, docker-compose files, CORS configuration, or scripts after a
    frontend or backend stack migration. Common markers include port numbers
    associated with the old stack (e.g., Vite :3000, :5173), service names
    matching the removed stack, and healthcheck paths that no longer exist.
  verify:
    - type: no_string_match
      files:
        - Dockerfile
        - docker-compose*.yml
        - docker-compose*.yaml
        - src/**/config.py
        - scripts/*.sh
      patterns:
        - "localhost:3000"
        - "localhost:5173"
        - "0.0.0.0:3000"
        - "/health/live"   # bare path (correct form is /api/health/live)
      note: >
        Patterns must be tuned per-project. After any stack migration, run a
        grep for the old stack's canonical ports and paths across all
        infrastructure artefacts.
  override_eligible: true
  override_reason_required: "Confirm old port/path is intentionally retained or document why the pattern is a false positive."
  added: "2026-05-08"
  source: "2026-05-08 audit NEW-001/003/004/005"
```

### SUP-001: Floating-tag supply chain risk

Applies to Dockerfile `FROM` lines, shell scripts that download binaries, and GitHub Actions workflow steps that reference `:latest` or a floating URL such as `releases/latest`.

```yaml
- id: SUP-001
  domain: supply-chain
  severity: suggested
  description: >
    Floating-tag or floating-URL dependency: a Dockerfile FROM instruction,
    shell script download, or workflow step references a floating tag such as
    `:latest` or a URL pattern ending in `/releases/latest`. These cannot be
    audited for known vulnerabilities because the resolved version changes
    silently over time.
  verify:
    - type: no_regex_match
      files:
        - Dockerfile
        - Dockerfile.*
        - scripts/*.sh
        - .github/workflows/*.yml
        - .github/workflows/*.yaml
      patterns:
        - ":latest"
        - "/releases/latest"
        - "releases/latest/download"
      exemptions:
        - "# allow-floating-tag: <reason>"
  remediation: >
    Pin to a specific version tag or SHA digest. For Dockerfile base images,
    use `image:version@sha256:digest`. For shell script downloads, record the
    pinned version in a comment and verify the checksum. For Actions, use a
    pinned SHA.
  override_eligible: true
  override_reason_required: "Document the specific tag and why pinning is not feasible, plus a tracking reference for follow-up."
  added: "2026-05-08"
  source: "2026-05-08 audit NEW-007/008"
```

### API-NEW-001: Sensitive identifiers in error responses

Applies to FastAPI (and any other HTTP framework) exception handlers where PII or regulated identifiers such as EIN, SSN, or account numbers are interpolated into the response detail string.

```yaml
- id: API-NEW-001
  domain: api-security
  severity: important
  description: >
    Sensitive identifier leakage in HTTP error responses: a regulated
    identifier (EIN, SSN, bank account number, or similar) is interpolated
    directly into an HTTPException detail string or equivalent error response
    body. The value is then returned to the caller and may appear in logs,
    error tracking tools, and browser network panels.
  verify:
    - type: no_regex_match
      files:
        - src/**/*.py
      patterns:
        - "HTTPException.*detail.*ein"
        - "HTTPException.*detail.*ssn"
        - "HTTPException.*detail.*account_number"
        - "raise.*HTTPException.*f\".*{.*ein.*}\""
        - "raise.*HTTPException.*f\".*{.*ssn.*}\""
      note: >
        Pattern matching is case-insensitive. Reviewers should also search for
        any f-string or .format() call inside an HTTPException detail argument
        that references a model field containing regulated data.
  remediation: >
    Replace the interpolated identifier with a generic reference. For example,
    replace `f"Entity with EIN {entity.ein} already exists"` with
    `"An entity with this EIN already exists"`. Return the conflicting
    resource's UUID (or a conflict URL) in a separate `headers` dict if the
    client needs to locate the duplicate.
  override_eligible: false
  added: "2026-05-08"
  source: "2026-05-08 audit NEW-014"
```

## Operational Lessons

### Lesson 1: Standards manifest absent -- agents ran without a shared check registry

Neither `~/.claude/docs/standards-manifest.yaml` nor any equivalent registry file existed at the time of this audit. Each domain agent independently discovered checks from direct inspection of the repo. This produced overlapping coverage in some areas (CORS config checked by both the foundations agent and the general auditor) and gaps in others (no agent checked for floating tags in scripts until the general auditor surfaced it as an unclassified candidate).

**Recommendation**: Create a minimal manifest stub at `~/.claude/docs/standards-manifest.yaml` that at minimum catalogs the check IDs used in this audit. Populate with the three new entries above. The manifest does not need to be exhaustive on first creation; even a 20-entry file allows agents to deconflict check ownership, avoid duplicate findings, and assign canonical severity levels consistently.

Concrete next step: copy the three YAML blocks from the "Proposed Manifest Additions" section above into the manifest file, then run the next audit with `manifest_version` set. Agents that load the manifest will skip any check already passing from a prior audit's remediation.

### Lesson 2: GitHub repos catalog absent -- OSSF agent queried APIs directly

`~/.claude/docs/reference/github-repos.json` does not exist. The ossf-compliance-auditor had to query the Scorecard API and Best Practices Badge API directly each run, which adds latency and creates a network dependency for a step that could be cached.

**Recommendation**: After each audit session, write a minimal JSON entry for the audited repo into `~/.claude/docs/reference/github-repos.json` capturing: repo slug, default branch, scorecard score at time of audit, badge percentage at time of audit, and date. This allows future runs to skip the API call when the data is fresh enough (configurable threshold, e.g., 7 days).

### Lesson 3: Parallel agent dispatch caused silent edit loss due to pre-commit auto-stash

The writing-style-editor agent edited 5 files but lacked Bash access to commit them. A parallel candidates agent then ran `git commit`, which triggered pre-commit's auto-stash. The auto-stash silently discarded the writing-style-editor's unstaged edits. Those edits were lost until the issue was diagnosed manually.

**Recommendation**: The coordinator must enforce a strict ordering rule: agents that produce file edits but cannot commit must be dispatched *before* any agent that invokes `git commit`. Alternatively, the coordinator should stage all pending edits immediately after each write-only agent completes, before launching the next agent. A safe pattern is: run all write-only agents sequentially, stage their output, then launch commit-capable agents. Never dispatch a write-only agent in parallel with a commit-capable agent.

If true parallelism is needed for throughput, each write-only agent should write to a named temp branch or a scratch directory, and a merge step should consolidate edits before any commit is attempted.

### Lesson 4: detect-secrets pin version mismatch creates silent schema incompatibility

The pre-commit hook uses SHA `847b6255c47c4cb1838b026934d950473c21c656`, which resolves to detect-secrets v0.14.4. The `.secrets.baseline` file generated by a contemporary `detect-secrets scan` uses the v1.5.0 schema. Running the v0.14.4 hook against a v1.5.0 baseline file causes the hook to exit with an error that looks like a scan failure rather than a version mismatch, making diagnosis non-obvious.

**Recommendation**: Bump the detect-secrets pin to a SHA that resolves to v1.5.0. After bumping, regenerate `.secrets.baseline` with `detect-secrets scan > .secrets.baseline` using the same pinned version. Add a comment on the SHA line in `.pre-commit-config.yaml` recording the version it resolves to (e.g., `# v1.5.0`), which is already required by PC-012 for all SHA rev lines.

### Lesson 5: CI audit findings must be verified against the live file before raising

CI-005 and CI-007 were raised against sonarcloud.yml at specific line numbers (129, 151, 164). At the time of the audit, that file had already been refactored from a full inline workflow into a 42-line thin reusable-workflow caller. The referenced lines do not exist. Both findings were marked N/A after manual inspection.

**Recommendation**: The ci-workflow-auditor agent should verify that the target file has at least as many lines as the highest referenced line number before raising a finding. If the file is shorter than expected, the agent should re-read the current file and either update the finding or suppress it with a note that the code path no longer exists. This is a one-line guard: `if actual_line_count < referenced_line: skip_finding()`.

More broadly, any agent that caches workflow structure from a prior scan and then raises findings against cached data should validate the cache against the live file's SHA before use.

### Lesson 6: Post-migration artefact decay is not caught by any existing check

When the React + Vite frontend was replaced by HTMX + Jinja2, four separate infrastructure artefacts retained references to the old stack: the Dockerfile healthcheck path, docker-compose.prod.yml structure, a stale service block in docker-compose.prod.yml, and scripts/generate-client.sh. A fifth artefact (CORS origins retaining Vite dev server ports) was identified but deferred. None of these were caught by existing domain agents because no agent checks for cross-artefact consistency after a migration.

**Recommendation**: Add a post-migration checklist to the project's CLAUDE.md or contributing guide. The checklist should enumerate all infrastructure artefacts that reference the frontend stack and require a sign-off for each when a migration PR is merged. Separately, the proposed FOUND-NEW-002 manifest check provides automated coverage for the most common patterns. Both measures are complementary: the checklist covers novel patterns, the manifest check covers known ones.
